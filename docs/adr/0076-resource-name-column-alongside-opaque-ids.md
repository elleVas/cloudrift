# ADR-0076: A `Name`/`User` column alongside opaque AWS-generated IDs, never replacing them

- **Status:** Accepted (2026-07-23)

## Context

Every report table's first column is the resource's identifier, and for most of the 43 `ResourceKind`s that identifier is an AWS-generated opaque string (`vol-…`, `i-…`, `snap-…`, `eni-…`) with no human meaning on its own. The user's explicit priority, prompted by comparing cloudrift's tables to a competitor's (kept internal — see [[feedback_no_competitor_mentions_public]] in memory, not repeated here): finding the exact resource on the real AWS console quickly matters more than any other report readability concern, and the ID is what gets pasted into the console search box, so it had to stay the reliable, unambiguous first column — a human-friendly name could only be *added*, never substituted.

Investigated before implementing rather than assumed: of the 18 `ResourceKind`s whose scanner didn't populate `tags` (hardcoded `tags: {}`), most already show a **user-chosen** identifier as their primary column (DynamoDB table name, Lambda function name, RDS-family `Identifier`, Log Group name, …) — adding a tag-based name there would have added little. The genuinely opaque-ID kinds (EBS volume, EC2 instance, snapshots, ENI, EIP, NAT Gateway, EFS, FSx, VPN connection, Transit Gateway attachment) already had real tags fetched by their scanners; the presenter simply never surfaced them. This reframed the actual scope from "fetch tags for 14 more scanners" down to a presenter-only change for 13 kinds, no new AWS API calls.

## Decision

`tagName(tags)` (`apps/cli/src/formatters/resource-presenters.ts`) returns `tags.Name || '—'`, inserted as the **second** column — right after the ID, never replacing it — in the 13 `ResourceKind`s with an opaque primary identifier: `ebs-volume`, `elastic-ip`, `ec2-instance`, `ebs-snapshot`, `nat-gateway`, `ebs-gp2-upgrade`, `ebs-idle`, `ec2-underutilized`, `eni-orphaned`, `efs-unused`, `fsx-idle-filesystem`, `vpn-connection-idle`, `transit-gateway-idle-attachment`.

`workspaces-idle` gets a `User` column instead of a `Name` tag: WorkSpaces are identified by their assigned user, not a name tag, and `DescribeWorkspaces` already returns `UserName` natively — a new `Workspace.userName` prop, populated from the existing API response, no new call. The other 30 kinds were left alone: already human-named, or the concept doesn't apply (ECR image digest, S3 multipart upload already showing bucket+key, `environment-ghost`'s synthetic resource group).

Because `resource-presenters.ts` is the single shared definition for the console table, the markdown formatter and the PDF formatter (`head`/`row` read directly, no per-format duplicate), all three picked up both new columns automatically — no changes needed there. The JSON export (`WasteReportDto`) already serialized the full `tags` object per finding regardless of kind, so `tags.Name` was already present in JSON before this change. The one real gap: `WasteReportDto.findings` only ever carries fields common to every `WastedResource` (never a kind-specific field — no `sizeGb`, no `instanceType` anywhere in it), so `userName` needed an explicit, **optional** addition — `isWorkspace()` type-guards `finding.kind === 'workspaces-idle'` and only that case populates `userName`; every other finding omits the key entirely rather than emitting `null`/`undefined` noise.

## Alternatives Considered

- **Fetch tags for all remaining 18 scanners, add `Name` everywhere (44/44).** Rejected after investigating: most of those 18 already show a human-chosen identifier, so the incremental value was low relative to the cost of 14 new per-service tagging API integrations. Left as a deferred option if the user wants full coverage later — the note is preserved in project memory for the next session, not written into this ADR as pending work.
- **A single centralized "fetch tags" function for all scanners.** Rejected: proposed by the user, investigated, and found not to centralize cleanly — every AWS service exposes a differently-shaped tagging API (`ListTagsForResource`/`ListTags`/`ListTagsOfResource`, some ARN-keyed, some name-keyed), so there is no single call to centralize. The only clean centralization point identified is the existing `CloudWatchIdleScanner` template method's optional-hook pattern (`resolvePrices()`-style), for the subset of scanners that already extend it — not exercised in this change since no new tag-fetching was needed after the rescoping above.
- **Replace the ID with the Name tag as the primary column.** Rejected outright per the user's explicit priority: the ID is the value that reliably finds the resource in the AWS console; a Name tag can be missing, stale, or duplicated across resources, so it is additive context, never the primary key.

## Consequences

Zero new AWS API calls — the 13 `Name` columns use tags already fetched, and WorkSpaces' `User` column uses a field already returned by `DescribeWorkspaces`. `WasteReportDto`'s `findings[].userName` is optional and present only for `workspaces-idle`, consistent with the DTO's existing generic-fields-only design rather than special-casing one kind into an otherwise uniform shape. Verified: lint clean, 438 aws-adapter + 105 cli + 24 application tests pass, build green across all 5 projects. `ResourcePresenter.colWidths` was noted as dead code during this work (the PDF formatter computes widths dynamically via `computeColumnWidths()`, never reads the static field) — updated for the 13 touched presenters for internal consistency, not removed from the interface; a future cleanup could drop the field entirely.
