# ADR-0082: MCP server as a second input adapter, not a fifth bounded context

- **Status:** Accepted (2026-07-24)

## Context

cloudrift has one input adapter today: the Commander-based CLI in `apps/cli`. The [Frontend-readiness](../en/architecture.md#frontend-readiness) section of `architecture.md` predicted, well before any code existed, that "a new entry point is just another composition root that instantiates the same [use cases] and calls the same `execute()`" — written with a future HTTP frontend in mind, but the same claim applies to any new entry point.

An [MCP](https://modelcontextprotocol.io) (Model Context Protocol) server turns that prediction into something immediately useful: AI coding agents (Claude Code, Kiro, VS Code Copilot Chat in Agent mode) can call cloudrift's four domains as tools directly, instead of a human running the CLI and pasting output back into the conversation. This was implemented — `mcp-server-application` (new lib), `mcp.composition.ts`/`mcp.command.ts` (new files in `apps/cli`), a `cloudrift mcp` subcommand, and an `e2e-localstack-mcp.mjs` harness — and verified end-to-end against both LocalStack and a real AWS account before this ADR was written.

## Decision

Add MCP as a second input adapter, reusing every existing composition root verbatim:

- **`mcp-server-application`** (new lib, `scope:application`): a single `AggregateAnalysisUseCase` that takes the four existing inbound domain ports as constructor dependencies (`FindWastedResourcesUseCasePort`, `FindDeadResourcesUseCasePort`, `FindResourceSecurityFindingsUseCasePort`, `CostTrendUseCasePort` — the same types `AnalysisContext.useCase` etc. already use elsewhere), runs them in parallel, and composes one report. It depends on domain ports only — never on any `*-application` package, AWS, or the MCP SDK — even though `@nx/enforce-module-boundaries`'s `scope:application → scope:application` constraint would allow the wider dependency; the narrower one is a deliberate choice to keep this lib as pure orchestration logic.
- **`apps/cli/src/commands/mcp.composition.ts`**: builds the four concrete, AWS-wired use cases by calling the exact same `defaultAnalyzeDeps`/`defaultDeadResourcesDeps`/`defaultResourceSecurityDeps`/`defaultCostAnalyticsDeps` objects the `analyze`/`dead-resources`/`resource-security`/`trend` commands already use — no parallel/duplicate wiring path.
- **`apps/cli/src/commands/mcp.command.ts`**: pure MCP protocol glue. Registers three tools on an `McpServer` (`@modelcontextprotocol/sdk`) with Zod input schemas — `analyze_cloudrift` (runs all four domains, real AWS calls), `get_resource_types` (static catalog), `get_required_iam_permissions` (static policy document, kept in sync by hand with `docs/en/iam-permissions.md`) — and exposes `mcpCommand()`, wired into `main.ts` as `cloudrift mcp`.
- **Partial-failure handling mirrors the existing `scanErrors` pattern.** `AggregateAnalysisUseCase.execute()` always returns `Result.ok(report)`; a domain whose own use case fails lands in `report.domainErrors` (`{ domain, message }` — flattened from `Error` because `JSON.stringify(new Error(...))` produces `{}`, no enumerable own properties) instead of failing the whole tool call. Same one-level-up version of the per-`(scanner, region)` `scanErrors` every domain already collects internally.
- **Credentials: no separate IAM role.** `cloudrift mcp` inherits the exact same AWS credentials as every other command. A `CLOUDRIFT_DISABLE_MCP` environment variable (checked before touching AWS credentials or reading any config file) is the documented kill switch for anyone who wants to be sure a given machine never starts the server — deliberately an env var, not a `cloudrift.config.json` flag, since `cloudrift mcp` works from any directory, with or without a project underneath it.

## Alternatives Considered

- **A separate, narrower IAM role/credential path for the MCP server.** Rejected: `cloudrift mcp` is a local process spawned by the MCP client with the operator's own environment — there is no server boundary to put a narrower role behind. Anyone who can configure the client can already read `AWS_PROFILE`/credentials directly; a second, more restrictive credential path would be security theater, not a real isolation boundary. The kill switch (disable the whole server) is the honest control surface instead.
- **Registering MCP tools inside each domain's own `*-application` package** (four separate tool sets) instead of one aggregate use case. Rejected: would leak the MCP protocol (or at least a protocol-shaped seam) into every domain's application layer, and would give a client four separate calls to make (and reconcile) for what is conceptually one account snapshot. One `AggregateAnalysisUseCase` in a new, protocol-agnostic lib keeps each domain's application layer untouched.
- **Do nothing / MCP as a documented-but-unbuilt possibility**, deferred like ADR-0020's multi-cloud path. Rejected: the value (agents calling scanners as tools instead of a human running the CLI) was concrete and immediately usable, and the composition-root reuse made the implementation cost low — this is the "new entry point is just another composition root" prediction being cheap to redeem, not expensive.

## Consequences

New `mcp-server-application` lib (`AggregateAnalysisUseCase`, its DTOs, unit tests). New `apps/cli/src/commands/mcp.composition.ts` and `mcp.command.ts` (plus `mcp.command.spec.ts`, an in-memory MCP client/server transport test covering tool registration, the aggregate report shape, partial-`domainErrors` pass-through, and the `CLOUDRIFT_DISABLE_MCP` kill switch). `main.ts` gained a `mcp` subcommand. `apps/cli/package.json` gained the `@modelcontextprotocol/sdk` dependency and an `e2e-localstack-mcp` Nx target backed by `scripts/e2e-localstack-mcp.mjs`. Documentation: the `mcp` command and its architecture are covered in `docs/en/usage.md`/`docs/it/utilizzo.md` and `docs/en/architecture.md`/`docs/it/architettura.md`; client-specific configuration lives in a dedicated doc — see [ADR-0083](0083-dedicated-mcp-client-config-doc.md).
